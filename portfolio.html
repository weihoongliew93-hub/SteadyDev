Offline POS Sync Optimization & Staging Database Innovation

Retail POS | Enterprise | Performance & Testing Enhancement

Offline-capable POS system relied on sync mechanism to synchronize data with server. Testing used fake outlets that generated noise data on production servers. Synchronous syncing (one table after another) made data downloads incredibly slow—when downloading the largest table, all other tables waited idle. Image downloads went through web service middleman, limiting throughput and consuming costly bandwidth.

Solution

Introduced staging local database with minimal coding effort by dynamically cloning all tables into separate copies through existing in-house Entity Framework middleware. Carefully orchestrated the middle layer to handle staging vs. production switching without changing a single line of application code. Enhanced sync mechanism with multi-threading support—smaller tables now download in parallel while large tables process, dramatically reducing total sync time. Redesigned image download to go directly to blob storage, bypassing web service middleman entirely.

Outcome

Staging database solved test data pollution problem while easing feature testing significantly. Multi-threaded sync transformed download performance—longest sync time now determined by single largest table instead of cumulative sequential processing. Direct blob storage downloads not only sped up image transfers but also reduced infrastructure costs by eliminating web service bandwidth consumption. Achieved major performance gains and cost savings through architectural improvements requiring minimal code changes.

.NET Multi-threading Entity Framework Azure Blob Storage Performance Optimization

Developer Productivity Framework Enhancements

Internal Tooling | Enterprise | Continuous Improvement

Development workflows suffered from fragile, maintenance-heavy patterns: printing module required modifying 4+ files for each new report; system settings jumbled together in a single file; enum-string list pairs maintained manually with high risk of sequence mismatches. Each pattern slowed development and introduced unnecessary risks.

Solution

Printing Module: Restructured system so adding new reports requires only 1 main file + 1-2 lines elsewhere—down from 4+ files. System Settings: Physically separated jumbled settings into module-specific files, then created framework that auto-generates UI from data models—new settings only need data model definition, no manual UI or save/load code. Enum-String Pairs: Refactored fragile positional dependencies into structured, readable code that eliminates sequence mismatch risks entirely.

Outcome

Printing enhancements: hours to minutes for new reports. Settings framework: adding new configuration became trivial instead of error-prone. Enum refactoring: completely eliminated maintenance risk and improved readability. Workflow problems were recognized early and reusable frameworks were built that benefit all developers. These improvements compound over time—every new feature leverages cleaner, safer patterns.
